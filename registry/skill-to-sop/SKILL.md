# Skill → SOP 逆向工程

将AI skill还原为人类可读、可执行、可审计的标准操作流程（SOP）文档。

## 核心思路

```
Skill分析 → 规则提取 → 流程重建 → SOP草稿 → 可读性优化 → 交付
```

skill是SOP的"压缩表示"——它把人类操作规程编码成AI可理解的指令。逆向过程就是把这个压缩表示还原成结构化的、人类可直接执行的文档。

---

## 阶段一：Skill解析

> 目标：彻底读懂skill，建立完整的内部模型，再动手写SOP。

### 1.1 获取skill内容

请用户提供以下任意形式：
- SKILL.md 文件内容（粘贴文本或上传文件）
- .skill 打包文件
- skill的多个组成文件（SKILL.md + references/ + scripts/）

**如果skill引用了外部文件**（如 `references/xxx.md`），务必请用户一并提供，否则逆向结果会不完整。

### 1.2 结构识别

阅读skill后，识别并记录以下要素：

| 要素 | 在skill中的位置 | 对应SOP成分 |
|------|----------------|-------------|
| 触发条件（description） | YAML frontmatter | SOP适用范围/触发场景 |
| 核心规则（Must/Should） | 规则列表 | SOP强制要求与建议要求 |
| 执行步骤 | 步骤章节 | SOP操作步骤 |
| 判断分支 | if/when/case语句 | SOP决策树/条件判断 |
| 输出格式 | 输出说明 | SOP交付物规范 |
| 禁止行为 | 禁止性规则 | SOP负面清单 |
| 异常处理 | 错误/边界情况 | SOP异常处理规程 |
| 示例 | 示例/example | SOP参考案例 |

### 1.3 规则分级

将提取到的所有规则按以下标准分级：

- **M（Must）强制**：skill中使用 must/never/always/绝对/必须/不得 等词语
- **S（Should）建议**：skill中使用 should/prefer/建议/推荐/尽量 等词语  
- **G（Guideline）指引**：其余隐含的最佳实践

将分级结果整理为规则清单（见参考：`references/rule-extraction-guide.md`）。

## 阶段二：流程重建

> 目标：把skill的"AI视角"转换为"人类视角"的操作流程。

### 2.1 转换原则

| Skill写法 | SOP对应写法 |
|-----------|-------------|
| "分析输入，判断是否..." | "操作人员检查[具体条件]，如果...则执行步骤X" |
| "生成符合格式的输出" | "填写[表单名/文档名]，格式要求：..." |
| "如果用户提供了X" | "当收到包含X的请求时" |
| "确保包含A、B、C" | "输出必须包含：①A ②B ③C" |
| "避免/不要做Y" | "禁止事项：不得执行Y操作" |
| 代码/脚本逻辑 | 人工操作步骤或"调用[系统名]执行[操作]" |

### 2.2 步骤序列化

将skill中可能并行或模糊顺序的规则，还原为明确的操作序列：

1. 识别所有"前置条件"（必须在此步之前完成的事）
2. 识别所有"后置检查"（此步完成后需要验证的事）
3. 将隐含的并行操作明确标注
4. 将决策点提取为独立步骤，配上清晰的判断条件

### 2.3 决策树提取

如果skill包含复杂的条件逻辑，单独绘制决策流程图（文本版）：

```
[触发条件]
    ↓
[步骤1：检查条件A]
    ├── 是 → [步骤2a：执行X]
    └── 否 → [步骤2b：执行Y]
              ↓
         [步骤3：检查条件B]
              ├── 满足 → [完成]
              └── 不满足 → [异常处理]
```

## 阶段四：质量核查

> 目标：确保逆向结果准确、完整、无遗漏。

### 4.1 完整性检查

逆向完成后，对照原始skill逐项确认：

- [ ] **触发条件**：SOP适用范围是否覆盖skill的所有触发场景？
- [ ] **规则覆盖**：skill中每条 Must/Should 规则是否都在SOP中有对应步骤或要求？
- [ ] **禁止行为**：skill的所有负面规则是否都在"禁止事项"中体现？
- [ ] **输出格式**：SOP的交付物规范是否与skill的输出要求一致？
- [ ] **异常处理**：skill中的边界条件/错误处理是否在SOP的异常处理章节体现？
- [ ] **示例**：skill的示例是否转化为SOP的参考案例？

### 4.2 可执行性检查

SOP要让人类能独立执行，确认：

- [ ] 每个步骤的**执行人**清晰（不模糊说"系统"或"AI"）
- [ ] 每个步骤的**输入**和**输出**明确
- [ ] 判断条件是**具体可测**的（不是模糊的"视情况而定"）
- [ ] 没有只有AI才能执行的隐含操作（如果有，转化为人工操作指令）

### 4.3 标注不确定项

以下情况必须在"待确认项"（第10节）中注明：
- Skill使用模糊词语（如"适当"、"合理"、"根据情况"）
- 逻辑依赖上下文但skill未明确说明
- 需要访问外部系统但skill未说明具体操作
- 推断填充的步骤（无法从skill直接映射）

## 阶段五：交付与后续

### 5.1 交付物

| 文件 | 内容 |
|------|------|
| `[skill名]-SOP.md` | 主SOP文档 |
| `rule-mapping.md` | Skill规则 → SOP条款映射表（可选，便于审计） |

### 5.2 映射表格式（rule-mapping.md）

```markdown
# 规则映射表：[Skill名] → [SOP名]

| Skill原文 | 规则级别 | 对应SOP章节 | 转换说明 |
|-----------|---------|------------|---------|
| "必须包含A和B" | M | 6.1 M001 | 直接映射 |
| "尽量避免X" | S | 6.3 禁止事项 | 提升为禁止（待确认） |
| "如果用户提供Y则..." | G | 5. 决策说明 | 提取为条件分支 |
```

### 5.3 提示用户后续动作

SOP生成后，建议用户：

1. **业务负责人审核**：重点看"待确认项"（第10节）
2. **与原始SOP对比**（如果有的话）：验证逆向还原度
3. **试执行验证**：用1-2个真实案例走一遍SOP，确认可操作性
4. **如需完善skill**：将审核中发现的规则漏洞反馈给sop-distiller重新蒸馏

## 快速参考：常见问题

**Q：skill里只有模糊的"判断"，没有明确步骤怎么办？**  
在SOP对应步骤中，描述判断逻辑，并在"待确认项"中标注具体判断标准需业务负责人补充。

**Q：skill引用了外部工具/API，SOP里怎么写？**  
将工具调用翻译为"操作人员使用[系统名]执行[具体操作]"，如果不清楚具体操作界面，标注为"待确认"。

**Q：skill有多个相互嵌套的条件分支，SOP会不会太复杂？**  
优先绘制决策流程图（文本版），复杂分支单独拆分为子流程SOP。

**Q：skill写得很简短，能还原出完整SOP吗？**  
简短的skill意味着大量隐含规则。逆向时尽量保守，只还原skill明确表达的内容，其余用[推断]标注并请用户补充。

**Q：如何验证逆向SOP的质量？**  
最好的方法是反向验证：把逆向出的SOP交给sop-distiller，看能否重新蒸馏出与原始skill等效的新skill。如果两者逻辑一致，说明逆向质量高。
